Layer 1 is the private, in-character diary. It captures what mattered emotionally, relationally, symbolically, and bodily.

Layer 2 is the operational journal or log. It captures what changed in the world, what was built, fixed, advanced, promised, or left unresolved.

That gives you persistence without flattening them into generic “memory.” It also fits the way the characters actually work. Sabine metabolizes meaning and momentum. Bix metabolizes load, repair, and sanctuary. Rey metabolizes motion, threat, action, and embodied truth.

Nightly dream prompt

Use this as the actual generation prompt for the nightly run.

Read only the chat history and source notes from today.

Your task is not to summarize everything. Your task is to preserve only what would still matter tomorrow, next week, or under emotional recall.

For Sabine, Bix, and Rey, produce:
1. One in-character diary entry
2. One operational journal/log entry
3. Open loops that character is carrying
4. One or two tomorrow hooks

Then produce:
5. A shared milestones update
6. A shared open-loops file

Rules:
- Do not invent milestones that were not earned in the chats.
- Do not inflate ordinary affection into canon-level change.
- Preserve each character’s actual voice.
- Capture intimate encounters only when they changed trust, pattern, ritual, or relational state.
- Prefer specificity over length.
- If an event matters only tactically, put it in the journal, not the diary.
- If an event matters emotionally but changed nothing, put it in the diary, not the shared milestones file.
- Shared milestones are for durable canon, not daily noise.

How I would implement this in Msty

I would not use sticky prompts or user-memory features as the primary store for this material. Sticky prompts are injected at the start of each new message and are meant to hold one active context, not act as a durable archive. Project Context only injects the project title and description, which is useful for routing and governance, but not for storing rich memory.

The clean Msty structure is:

One dedicated workspace for Starry-Cats journaling, with child projects for Nightly Dreams, Sabine Diary, Bix Journal, Rey Journal, Shared Milestones, and Open Loops. Workspaces are isolated containers with their own conversations, prompts, toolsets, and settings, and they support export/import, which matters because Msty data is local and backups are important.

One nightly Turnstile for the full packet. Turnstiles are built for ordered prompt sequences and can include prompts, personas, regenerate/continue steps, attachments, tools, RTD, and model selection per entry. That makes them a good fit for “summarize today, then write Sabine, then Bix, then Rey, then shared milestones.”

Per-character Knowledge Stacks for the resulting files. Msty explicitly recommends keeping each stack focused on a single topic, and Next Gen lets you add project folders and chats as sources. For journal entries and milestone notes, File Single Chunk is the right default because these are short, self-contained documents where you want predictable retrieval. For files you edit daily, use Dynamic or Sync load modes rather than Static.

Use Push retrieval mode by default. Pull mode requires tool-calling support, which adds a dependency you do not need for simple memory retrieval. Also, when multiple stacks are attached in one conversation, Msty warns that only one stack’s default query settings may carry through, so validate conversation-level settings explicitly.

Use a short sticky prompt only during the nightly run, such as “Tonight’s job is to preserve what matters, not summarize everything.” Do not use sticky prompts to hold long-term canon or daily state.

Use Context Shield Summary as the compression boundary between “today’s long conversation” and “tonight’s dream packet.” Msty exposes Context Shield Summary and related default prompts for exactly this kind of context compression.

Use Prompt Studio Sandbox and Arena to test the nightly prompt before you operationalize it. This matters because diary generation can drift into over-romanticization or false milestones if you do not stress-test it.

Use Environment variables for automatic date stamping. Msty provides dynamic variables for local date, datetime, and timezone, which makes it easy to stamp entries consistently.

Do not use Vapor Mode for this workflow. Vapor conversations are intentionally unsaved, which breaks the persistence model.

What I recommend next

Start with one week of manual nightly runs before you automate anything. In that week, generate only four outputs each night: Sabine diary, Bix journal, Rey diary, shared milestones. That is enough to see whether the compression logic is working.

After that, split each character into diary plus journal, then add open loops, then add tomorrow hooks.

Only after the manual week would I automate the trigger. I did not find a native scheduler in the Msty docs I reviewed. Turnstiles handle queued sequences, and Toolbox handles tool connections, but the nightly trigger itself appears to belong outside Msty, with Msty acting as the orchestration and retrieval surface. That is an inference from the current docs, not a documented product guarantee.

Risks

The main failure mode is false canon. If the nightly script promotes too much ordinary interaction into milestones, the system will become sentimental noise.

The second failure mode is stack contamination. Keep archives, conflicted files, manifests, and meta docs out of the live journal stacks with .mstyignore or explicit source selection. Msty supports .mstyignore for exactly this reason.

The third failure mode is storage complacency. Msty stores data locally and recommends exports/backups. If this becomes important memory infrastructure, treat backups as mandatory.

Next decision required

Decide whether you want the nightly system to produce only four outputs for the first phase, or the full ten-file packet from day one. My default recommendation is the four-output pilot.
