F1 (HIGH) — AC-10.26 violation: scene_fodder never written to open_loops.
  - generate_consistency_qa() returned scene_fodder for healthy_divergence
    but neither the generator nor the runner called write_new_open_loops()
    with source="dreams_qa". The Step 9 execution record falsely claimed
    PASS; the writer signature was wired but no call site existed.
  - Fix: added runner._route_qa_scene_fodder_to_open_loops() that, after
    the QA pass completes, iterates HEALTHY_DIVERGENCE checks and writes
    one OpenLoop row per non-empty scene_fodder string. Anchored on
    relationship.pov_a so the row surfaces in either character's
    open-loop scan; pov_b lands as best_next_speaker. The writer's
    source="dreams_qa" qualifies loop_type to "dreams_qa_scene_seed:dreams_qa".
  - Routing wrapped in try/except so failures become warnings, not crashes.
  - New integration regression at
    tests/integration/test_dreams_consistency_glue.py
    ::test_healthy_divergence_scene_fodder_lands_in_open_loops asserts
    rows land tagged correctly with the right counterpart speaker.

F2 (MEDIUM) — Markdown ledger race condition.
  - _emit_markdown did read_text() → in-memory replace() → write_text()
    with no file lock. Two concurrent Dreams runs on the same UTC date
    (cron jitter / manual replay / systemd retry) would clobber each
    other.
  - Fix: added _file_lock(handle) cross-platform context manager
    (msvcrt.locking on Windows, fcntl.flock on POSIX, no-op fallback
    with warning on stripped builds). Refactored _emit_markdown to open
    "r+", hold the lock for the full read-modify-write critical section,
    re-read inside the lock, truncate + write + fsync before releasing.
  - Extracted _render_entry() helper to keep the lock-protected section
    focused.

F3 (MEDIUM) — Auto-promote test name read backward.
  - Renamed test_should_promote_returns_true_at_exactly_threshold_minus_one
    to test_should_promote_triggers_when_tonight_completes_threshold_chain
    with an explanatory docstring noting that THRESHOLD_NIGHTS=3 means
    "promote on the 3rd consecutive night" — should_promote is consulted
    ON the contradicting night, so seeing 2 prior qualifying nights is
    the trigger.

Phase 11 — Cross-Persona Context Injection (SHIPPED 2026-04-17)

HIGH-severity gap surfaced while authoring Docs/MSTY_SETUP.md.

Bug: pipeline.run_chat_pipeline extracted Msty Crew Conversations
prior-persona responses (MstyPreprocessed.prior_responses) for scene
classification but never injected the actual response text into the
focal persona's outbound user_prompt. PHASE_7.md §R2-F1 had documented
a planned _format_crew_prior_block helper; the helper closure note was
written but the helper was never shipped into pipeline.py. Combined
with Msty Studio's actual Crew Contextual mode behavior (per
docs.msty.studio/conversations/crew-chats step 5: "Contextual where
they are aware of persona responses before theirs"), the operator's
expected user experience was structurally impossible: each persona
answered as if speaking alone, with no riff or acknowledgment of the
prior persona's text.

Fix: add _format_crew_prior_block(prior_responses, current_user_message)
to pipeline.py + wire-in inside run_chat_pipeline. Frame:

  [Earlier in this conversation:
  **<character_id>:** <html-escaped, 800-char-capped text> [...truncated]?
  ]

  <current_user_message>

When prior_responses is empty, helper returns current_user_message
unchanged — Phase H regression byte-identical for non-crew flows.
HTML-escape + per-block truncation = Phase 8 R1-F3 lesson applied.

continues to receive focal-only full_response_text. The augmented
user_prompt is request-side only — dyad deltas score the focal
persona's output, not the conversation as a whole. Mixing prior
personas' text into dyad updates would conflate other characters'
speech into the focal character's deltas. Documented as deferred
operator-tools-phase consideration.
